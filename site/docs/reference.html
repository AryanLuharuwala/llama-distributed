Design notes

The original framing, the tradeoffs, and what's still open.

Motivation

llama.cpp started as a single-node server or CLI to run inference on local machines. It operates on GGUF files — quantized or full-weight models, trained or fine-tuned with tools like Unsloth.

The issue with multi-host rollouts is that they are generally distributed over a network switch rather than over the internet. The switch provides fast transfer using NCCL or MPI, but there's no complete story for moving packets over secure public channels.

llama-distributed solves that with TCP transfer wrapped in WebSocket relays, peer to peer. A coordinator handles signaling, relays, and scheduling of the nodes based on VRAM availability (and token usage — TODO), to ensure each GPU sustains maximum useful compute.

Heterogeneity

Some GPUs speak Vulkan, some Metal, some CUDA. The protocol has to be platform-agnostic on Windows, macOS, and Linux, handled by a single web interface that manages the pool.

The llama_decode public API uses the graph generated during model registration and supports model-graph splitting so only a few layers run on any single node. The first node holds the tokenizer and token embedding; the last node holds the output embedding.

Objectives

  • llama_decode runs configurable layer ranges instead of requiring hardcoded layer assignments — fully configurable graph.
  • Coordinator schedules layers so utilization is equated across nodes; falls back to idle nodes when more than one hosts the same layers.
  • DNS-like system (Vercel-style) so each pool gets its own OPENAI_BASE_URL. Bearer-token auth for AI requests, email-based pool ownership.
  • TURN / STUN server for NAT'd rigs — P2P connections between networks behind firewalls.
  • Intuitive Web UI — users connect their own server in under 4 clicks.
  • Single download script (curl install.sh | bash, .dmg, .msi) that auto-detects OS, installs the program, and has the dashboard generate a deep-link URL that opens in the native installer.
  • Deep link connects the backend to the server; the frontend is disentangled from using a localhost address so it fetches live data from our public server.

Open questions

  • Partial model loading — assign weight tensors per node via tensor_buft_overrides. Today, every node loads the full GGUF and only uses its assigned layers. Wasteful for large quantized models.
  • Pre-connected data sockets — connect node pairs at startup so the first tensor hop doesn't pay a TCP handshake.
  • Multi-request interleaving — batched decode across concurrent sessions to push utilization when one request stalls.
  • RDMA transport — ibverbs / UCX for InfiniBand deployments.
  • Python bindings for VmContext.
  • Web dashboard improvements — node status, VRAM, throughput via HTTP/SSE.

Build script shorthand

scripts/build.sh
./scripts/build.sh [flags]

  --cuda            Enable CUDA
  --metal           Enable Metal
  --vulkan          Enable Vulkan
  --debug           Debug build with symbols
  --jobs N          Parallel build jobs (default: nproc)
  --prefix PATH     Install prefix (default: ./install)
  --llama PATH      External llama.cpp source tree instead of the submodule

Roadmap

  • Partial model loading — per-node weight assignment via tensor_buft_overrides
  • Pre-connected data sockets — connect node pairs at startup
  • Multi-request interleaving — batched decode across concurrent sessions
  • RDMA transport — ibverbs / UCX for InfiniBand
  • Python bindings for VmContext
  • Web dashboard — node status, VRAM, throughput via HTTP/SSE
  • TURN / STUN for NAT traversal on public deployments
