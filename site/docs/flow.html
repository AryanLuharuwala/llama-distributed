End-to-end flow

From clicking Add rig to tokens streaming out. Every step, every friction point.

The happy path

  1. User opens https://apex, signs in with GitHub OAuth.
  2. Clicks + Add rig. Dashboard picks a pool to auto-join from the dropdown.
  3. Server mints a 5-minute pair token, returns a bash and a pwsh one-liner.
  4. User pastes on the rig. Installer detects OS + accelerator, downloads the tarball, registers a service.
  5. dist-node opens a WebSocket, sends the pair token, receives a permanent agent_key.
  6. Rig shows up green in the dashboard, automatically attached to the pool.
  7. User clicks + New key. Server generates sk-dist-…, returns it exactly once.
  8. Any OpenAI SDK pointed at https://<slug>.<apex>/v1 with that bearer routes to the pool.

Step-by-step: what each click does

Login

  • GET / serves the embedded SPA.
  • GitHub OAuth round-trip in prod; -dev flag exposes POST /auth/dev for local work.
  • Session cookie is an HMAC-signed user id.

Generate install command

  • Browser → POST /api/install_command.
  • Server mints a 20-byte pair token, TTL 5 min, stores it in pair_tokens with user_id.
  • Response has two ready-to-paste strings — bash one-liner and pwsh equivalent.
  • UI tabs between the two based on navigator.platform.

Install on the rig

  1. install.sh detects OS + arch + accelerator (nvidia-smi → cuda; darwin-arm64 → metal).
  2. Resolves https://github.com/<repo>/releases/latest → tag.
  3. Downloads llama-distributed-<tag>-<target>.tar.gz.
  4. Extracts to ~/.local/share/llama-distributed (linux) / ~/Library/Application Support/llama-distributed (mac) / %LOCALAPPDATA%\llama-distributed (win).
  5. Writes a per-user service unit (systemd / launchd / Scheduled Task).
  6. Execs dist-node --pair <url>.

First-time pair handshake

  • dist-node dials the WebSocket URL from the deep link.
  • Sends { "type": "pair", "token": "…", "hostname": "…", "gpus": […] }.
  • Server calls consumePairToken(token) (single-use, TTL-guarded), inserts a row in rigs with a fresh random agent_key, sends { "type": "paired", "rig_id": N, "agent_key": "…" }.
  • Agent persists agent_key to a local config file.

Subsequent reconnects

  • dist-node dials /ws/agent, sends { "type": "auth", "agent_key": "…" }.
  • Server looks up the rig, marks online, subscribes it to events.
  • Reconnect with exponential backoff when the WS drops.

Create a pool

  • Browser → POST /api/pools {name, visibility}.
  • Server runs pickPoolSlug(userHandle, name) — slugify + collision check — and inserts.
  • Returns {id, slug, base_url}. In prod: https://<slug>.<apex>/v1. In dev: http://host/v1/<slug> (no wildcard DNS needed).

Attach a rig to a pool

  • Browser → POST /api/pools/{id}/rigs {rig_id}.
  • Server writes pool_rigs row, pushes a pool_rigs_changed event down every open /ws/browser for the pool's members.

Mint API key

  • Browser → POST /api/api_keys {label}.
  • Server generates sk-dist-<48 hex>, stores the sha256 hash plus a 12-char plaintext prefix for display. Plaintext returned exactly once.
  • List endpoint never reveals the full key — only prefix + metadata.
  • Revoke = DELETE /api/api_keys/{id}.

Inference request

  • Client hits POST https://alice-alpha.apex/v1/chat/completions with Authorization: Bearer sk-dist-….
  • resolvePoolFromHost() parses <slug> from r.Host. Fallback for LAN/dev: X-Pool-Slug header → ?pool=<slug> query param.
  • authOAI() hashes the bearer, looks up the api_keys row, checks pool membership, bumps last_used_at.
  • Server picks an online rig in the pool, forwards the request over its WS, streams the response back as SSE (OpenAI chunk format + [DONE]) or buffered JSON.

Known friction (and the plan to remove it)

Rated by how much the user has to do, not by eng effort.

Installer depends on a public GitHub release

Friction: the one-liner 404s until we cut a tagged release.

Plan: serve dist-node tarballs directly from the control plane at /releases/<target>.tar.gz. The installer already accepts --repo / --version.

Pair token expires in 5 minutes

Plan: bump TTL to 30 min for install-command tokens, or detect expiry on the install page itself and auto-regenerate on click. expires_at is already in the response.

Attaching a rig to a pool is a second click

Plan: add a "Pool to auto-join after pairing" dropdown on the Install card. Server accepts an optional pool_id in the token payload; inserts pool_rigs atomically after pair. Shipped in v0.3.

Firewall blocks port 8080 on self-host

Plan: print a "sharing" pre-flight check on startup. Try listening, try dialing ourselves, log the exact firewall-cmd line to paste. Shipped in v0.3.

No "copy OpenAI snippet" for the pool

Plan: on the pool detail modal, add a "Use with" subsection with collapsible code blocks for Python / Node / curl. All pre-filled with base URL and a placeholder API key.

The zero-friction flow (target)

  1. Open https://apex. Sign in.
  2. Click Add rig. Pick the pool to auto-join. One-liner appears.
  3. On the rig: paste. Installer runs, agent auto-pairs, auto-joins.
  4. Back on dashboard: rig is green, in the pool.
  5. Click Get OpenAI snippet. Paste into your code. Ship.

Five clicks, one paste, one copy. No firewall surgery, no SSH, no token juggling, no CI waits.
