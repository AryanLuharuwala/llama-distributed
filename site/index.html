Run any LLM across any pool of GPUs.

llama-distributed splits models layer-by-layer across a cluster of heterogeneous GPU rigs, streams activations between nodes, and exposes the whole pool as a single OpenAI-compatible endpoint. No tensor-wise sharding dance, no custom SDK.

Get Started View on GitHub
Stable v0.3 CUDA / Metal / Vulkan / CPU OpenAI SDK drop-in
client.py 
# point any OpenAI SDK at your rig pool
from openai import OpenAI

client = OpenAI(
    base_url="https://alice-alpha.dist.dev/v1",
    api_key="sk-dist-…",
)

resp = client.chat.completions.create(
    model="llama-3-70b",
    messages=[{"role": "user",
               "content": "Explain ring-AllReduce."}],
    stream=True,
)
for chunk in resp:
    print(chunk.choices[0].delta.content, end="")
Why it's different

Built for heterogeneous GPU pools, not just homogeneous clusters.

Four guarantees that make llama-distributed viable as the shared inference layer for any team.

Layer-split distribution

Split a 70B model across a 24 GB laptop and a 40 GB workstation. VRAM-proportional planner picks the cuts.

OpenAI SDK drop-in

Every pool gets a base_url and an API key. Existing OpenAI code works unchanged.

Multi-accelerator

Mix CUDA, Metal, Vulkan, and CPU rigs in the same pool. The protocol is platform-agnostic.

Zero-friction onboarding

One-liner installs, WebSocket auto-pair, pool auto-join, built-in firewall preflight.

      How it works
      
A control plane and a pool of dumb workers.

      
The Go server handles auth, pool routing, and OpenAI-compat endpoints. C++ rigs just run their assigned layers.

      Single-machine demo
      
Test the whole pipeline on one laptop.

      
Three processes on localhost: coordinator, two nodes, client.

    

    

      

        

          Terminal 1 · Terminal 2 · Terminal 3
          
        


# build (takes ~5 min for llama.cpp)
./scripts/build.sh --cuda

# 1. coordinator (terminal 1)
./build/dist-coordinator --model model.gguf --min-nodes 2

# 2. two node agents (terminals 2+3)
./build/dist-node --server 127.0.0.1 --data-port 7701
./build/dist-node --server 127.0.0.1 --data-port 7704

# 3. send a prompt (terminal 4)
./build/dist-client --server 127.0.0.1 \
    --prompt "Explain ring-AllReduce."

      

      

        
What you'll see

        
Tokens stream to stdout as they're generated. The coordinator picks layer assignments from VRAM, the two nodes pair up, and activations hop between them.

        
    
          
  • Cluster boots in < 5 seconds on a warm cache.
    • 
          
  • Pipeline overhead is dominated by NIC latency, not GPU compute.
    • 
          
  • --min-nodes blocks inference until the quorum is green.
