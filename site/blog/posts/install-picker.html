Building a PyTorch-style install picker for a C++ project

Apr 18, 2026 · Engineering · 5 min read

If you've ever installed PyTorch, you've used their picker: five buttons that combine into one command. It's such a small piece of UI that it's easy to miss how much work it's doing — guarding against 18 impossible combinations, generating the right --index-url for every CUDA version, handling the preview-nightly channel.

We wanted the same thing for llama-distributed. Our matrix is bigger (Build × OS × Role × Accelerator × Method = 2 × 3 × 3 × 4 × 3 = 216 combos), but the constraint was the same: the user clicks, the command is correct, no backend round-trip.

The shape of the picker

Five axes:

AxisValues
BuildStable · Nightly
OSLinux · macOS · Windows
RoleRig (agent) · Server (control plane) · Client
AcceleratorCUDA · Metal · Vulkan · CPU
MethodOne-liner · Tarball · From source

Disabling impossible combinations

Metal on Linux doesn't exist. CUDA on macOS doesn't exist since Apple dropped NVIDIA. Our first take on this was to hide invalid options — but that's worse than leaving them visible and struck through. Users lose their mental model of "OS has 3 options" if options appear and disappear.

installer.js
function combosValid(os, accel) {
  if (os === "linux")   return accel === "cuda"   || accel === "vulkan" || accel === "cpu";
  if (os === "macos")   return accel === "metal"  || accel === "cpu";
  if (os === "windows") return accel === "cuda"   || accel === "vulkan" || accel === "cpu";
  return false;
}

When OS changes, we check the current accelerator. If it's now invalid, we coerce to a sensible default (CUDA for Linux/Windows, Metal for macOS) rather than leaving the user in an error state.

Colorizing the command

The generated command lives in a <pre> with syntax-highlighted flags, strings, and comments. No syntax highlighter, no parser — just three regexes:

installer.js
function colorize(cmd) {
  return cmd.split("\n").map(function (line) {
    if (/^\s*#/.test(line))
      return '<span class="tok-cmt">' + escapeHtml(line) + '</span>';
    var html = escapeHtml(line);
    html = html.replace(/(--?[A-Za-z][A-Za-z0-9-]*)/g,
                        '<span class="tok-flag">$1</span>');
    html = html.replace(/('[^']*'|"[^"]*")/g,
                        '<span class="tok-str">$1</span>');
    return html;
  }).join("\n");
}

This misses cases — escaped quotes inside strings, flag-like tokens inside comments. But it's 15 lines of JavaScript and covers every real-world command we generate. We didn't want to ship a tokenizer library for this.

The edge cases that matter

A good picker acknowledges the combos it can't fix. Three notes appear dynamically:

  • Warn when you pick CUDA + macOS: "CUDA not available on macOS — switch to Metal or CPU." We still generate the closest command, but the warning prevents a 5-minute detour.
  • Info when you pick Server + any accelerator: "The control plane is pure Go — the accelerator setting only affects compute rigs." Users expect the Accelerator axis to do something for every role. It doesn't for the server. Saying so inline is cheaper than writing a docs page.
  • Info on Nightly: "Tracks the main branch. Expect occasional breakage." We learned not to assume users read release notes.

Prereq line

Below the command box, a muted one-liner lists the minimum requirements for the current combo:

  • CUDA → "NVIDIA driver 535+ · CUDA 12.5"
  • Metal → "macOS 13+ on Apple Silicon"
  • Source → adds "CMake 3.14+ · GCC 11+ or Clang 14+"

This is the smallest amount of text that answers "why is this failing on my machine?" before the user hits paste.

The copy button

PyTorch's picker has a copy button. We added one too, because the alternative — selecting multi-line text with the mouse — is surprisingly fiddly. The gotcha: navigator.clipboard.writeText() needs a secure context (HTTPS or localhost). On a raw-HTTP self-hosted dashboard over LAN, it silently fails.

Workaround: a hidden <textarea> with the raw command, with document.execCommand('copy') as fallback. The fallback is deprecated but still works everywhere we need it.

What we'd do differently

  1. Ship it earlier. We spent a week on the picker. It paid for itself the first time a user installed on Windows without asking us what to paste.
  2. Don't make axes interactive until they're needed. Our first version hid the Role axis behind an "Advanced" toggle. Turns out the first question everyone asks is "am I installing a server or a rig?" — so Role is now a top-level axis.
  3. The command is the source of truth, not the state. We initially stored the state as an object and rendered the command from it. That works. But it meant every command change required a UI re-render. It's cheap enough; we don't regret it. But if we were doing it again, we'd generate commands as pure functions and cache.

The picker is live on the homepage and the install guide. Source in site/assets/js/installer.js.
