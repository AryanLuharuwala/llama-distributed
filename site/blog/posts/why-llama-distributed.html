Why we built llama-distributed

Apr 02, 2026 · Intro · 6 min read

The 70B Llama weights don't fit on a 24 GB laptop. They also don't fit on most 40 GB workstation cards. But they do fit on two cards — or on a 40 GB card plus a 24 GB card, if you're willing to split the model across both.

The question is how you split.

Two ways to split a model

Tensor parallelism slices each weight matrix across devices. Every GPU participates in every layer; they AllReduce after every attention and MLP block. This is what tensor_parallel_size=2 in vLLM does. It's fast, and it's what everyone uses at scale.

But AllReduce needs serious interconnect bandwidth. Between two A100s on the same host with NVLink, you get 600 GB/s. Between two consumer cards over PCIe, you get ~32 GB/s. Between two cards on different machines over a 10 GbE switch, you get 1.25 GB/s. Between two cards over an actual network, you get whatever the network gives you — which for most home labs is 100 MB/s at best.

Tensor parallelism over a slow link is an anti-pattern. The AllReduce dominates.

Pipeline parallelism — or, more specifically, layer-parallel — slices the model by layer instead of by tensor. Node 0 owns layers 0–19, node 1 owns 20–39, node 2 owns 40–79. Each node runs llama_decode() on its slice, hands the activation tensor to the next node, and waits.

The network traffic is the activation tensor itself — maybe 4 KB for a single token of a 4096-dim model. That fits in a single Ethernet frame. Over a 1 GbE home switch, your per-token latency is ~0.04 ms of network time. Over 100 Mbps WiFi, it's ~0.4 ms. Either way, the GPU is the bottleneck, not the network.

The tradeoff

Pipeline-parallel is worse for a single request with no batching: only one GPU is busy at a time. But:

  • With double-buffering, a node's RX thread queues up the next activation while its compute thread runs the current one. GPU compute overlaps with NIC receive.
  • With request interleaving, multiple inference requests flow through the pipeline simultaneously — different tokens are on different nodes at the same time. Utilization approaches 100% under realistic load.
  • Tensor-parallel never achieves more utilization than the slowest GPU × the AllReduce overhead. Pipeline-parallel can.

We bet on pipeline-parallel because it's the only approach that works over the kind of hardware real people have — a home office with two gaming PCs, a university lab with mismatched workstations, a small company with three on-prem machines and no InfiniBand budget.

The heterogeneity problem

Tensor parallel assumes all devices are identical. If you split a 10,000 × 10,000 matrix evenly across two GPUs and one is a 4080 and the other is a 3060, the 4080 waits for the 3060 at every AllReduce. You're pinned to the weakest link.

Pipeline parallelism can be asymmetric. A 40 GB node can host 40 layers; a 20 GB node can host 20. We don't divide work evenly — we divide it proportionally. The planner reads each rig's reported VRAM, sums the total, and carves layer ranges by ratio. The rigs process at their own pace; the pipeline fills up and drains naturally.

This is why our pool can mix a CUDA 4090, a Metal M3 Max, and a Vulkan RX 7900 in the same cluster. None of them have NVLink to each other. None of them need to.

Why now

Two things converged in 2025:

  1. llama.cpp matured. The llama_decode() API is stable across backends. GGUF is the de-facto quantization format. CUDA, Metal, and Vulkan all compile from the same source tree. We don't have to pick a platform — llama.cpp already solved that.
  2. Home labs grew. A used 3090 is $650. A new 4090 is $1500. A Mac Studio with 128 GB of unified memory is $4000. People have real VRAM at home now, often spread across multiple machines they already own. The bottleneck is pooling, not silicon.

If you have the hardware and you're willing to split the model, you should be running a 70B model today. llama-distributed tries to be the shortest path from "I have some GPUs" to "my code talks to an OpenAI-compatible endpoint that runs a 70B model."

What llama-distributed is

  • A Go control plane (dist-server) that runs on a laptop, handles auth, assigns pair tokens, and exposes OpenAI-compat endpoints per pool.
  • A C++ node agent (dist-node) that runs on each GPU host, loads its assigned layer range, and streams activations to the next node.
  • A wire protocol so boring it's reimplementable in an afternoon in any language that has TCP sockets.
  • An installer that actually works, including the awkward cases (Windows, behind NAT, over LAN).

What it isn't

  • A replacement for vLLM or TGI at production scale. If you have an H100 cluster with NVLink, keep using what you're using.
  • A training framework. Fine-tune with Unsloth, quantize with llama.cpp's tools, then deploy here.
  • A TEE or zero-trust system. Assume a trusted LAN, or terminate TLS upstream (Caddy, Cloudflare Tunnel, Tailscale).

The goal is to make distributed inference work for the 80% of users who don't have an HPC cluster. If that's you, install it and see how far you get. File an issue when something breaks — there's a lot left to cut down.
