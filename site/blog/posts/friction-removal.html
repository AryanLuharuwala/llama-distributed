Friction-removal pass: path-form pools, auto-join, preflight

Apr 22, 2026 · v0.3 release notes · 4 min read

v0.3 was supposed to be a maintenance release. Instead, we spent a week cutting friction from the onboarding flow — three small changes that turn a 10-click setup into a 3-click one. None of them are architecturally interesting. All of them are the difference between a user getting to their first token in 45 seconds or giving up after 5 minutes.

1. Path-form pool URLs

Previously, every pool got a subdomain: alice-alpha.dist.dev/v1. This is the clean form, and it still works in prod. But it requires wildcard DNS, which requires a wildcard TLS cert, which means Caddy / Cloudflare / whatever you're terminating with actually has a wildcard cert for the apex. Most self-hosters don't.

So v0.3 accepts /v1/<slug>/… as a synonym:

before (prod only)
client = OpenAI(base_url="https://alice-alpha.dist.dev/v1", ...)
after (LAN, self-host, or prod)
client = OpenAI(base_url="https://dist.dev/v1/alice-alpha", ...)

Resolution is simple: resolvePoolFromHost() first tries the subdomain form, then falls back to the first path segment after /v1/. Existing subdomain URLs keep working — this is additive.

We also kept the ?pool=<slug> query-string form as a last-resort fallback, but many SDKs strip query strings when they append paths like /chat/completions. Path form is now the recommended dev URL.

2. Pool auto-join on pair

The old flow:

  1. Click Add rig → paste one-liner → rig pairs.
  2. Click New pool → name it → create.
  3. Find the rig in the list → drag it into the pool.

Three clicks, two of which don't need to exist. The dashboard already knows what pool you want the rig in — you either have one, or you're about to make one. So the Add rig card now has a "Pool to auto-join" dropdown that defaults to your only pool (if you have one) or None.

Server-side, the pair token payload picks up an optional pool_id. On successful pair, we insert the pool_rigs row atomically before we reply paired. The browser gets a pool_rigs_changed push and the rig shows up already green, already in the pool. One click becomes zero.

3. Firewall preflight check

We hit this the night before the release: on a fresh LAN setup, the control plane binds to 0.0.0.0:8080, everything looks fine in the logs, and then the rig across the room prints connection refused. Windows Defender was silently blocking the port.

Error messages in distributed systems are famously bad at saying where a connection fails. We added a one-shot sanity check at startup:

startup log
2026-04-22 18:03:12 listen: http://0.0.0.0:8080
2026-04-22 18:03:12 preflight: dialing self at http://192.168.1.42:8080 ...
2026-04-22 18:03:13 preflight: FAILED — connection reset
2026-04-22 18:03:13 preflight: Windows Defender Firewall is blocking inbound 8080.
                               To open it:
                               New-NetFirewallRule -DisplayName "dist-server" -Direction Inbound -Protocol TCP -LocalPort 8080 -Action Allow

Same for firewalld on Linux (firewall-cmd --add-port=8080/tcp --permanent) and pf on macOS. The error moves from "why can't my rig reach the dashboard" to "copy this line and paste it". The underlying problem is the same; the user's time-to-fix drops from 20 minutes to 20 seconds.

Net result

The zero-friction path is now:

  1. Open https://apex. Sign in.
  2. Click Add rig (pool auto-selected). Copy one-liner.
  3. Paste on the rig.
  4. Click Get OpenAI snippet. Paste into your code.

Four clicks total, two pastes. Full details in the flow doc.

What's next

  • QR-code one-liner so phones-next-to-rigs work without SSHing.
  • Installer served by the control plane — cut the GitHub Releases dependency for truly air-gapped deployments.
  • Inline OpenAI snippet on the pool modal (Python / Node / curl) with the API key pre-filled.
